# Token 消耗分析报告

## 一、当前调用情况

| 功能 | 文件 | 模型 | 输入内容 | max_tokens | 调用时机 |
|------|------|------|----------|------------|----------|
| 简历优化 | resumeService.js | **gpt-4** | 岗位描述全文 + 简历全文 + 长提示词 | 2000 | 用户点击「优化简历」 |
| 推荐信生成 | coverLetterService.js | **gpt-4** | 岗位描述全文 + 简历全文 + 长提示词 | 1000 | 用户点击「生成推荐信」 |
| 职位信息提取 | jobInfoExtractor.js | **gpt-4** | 岗位描述全文 + 提示词 | 200 | 粘贴岗位描述时（自动） |

一次完整流程（上传简历 → 粘贴岗位描述 → 优化简历 → 生成推荐信）至少触发：
- **1 次** 职位信息提取
- **1 次** 简历优化
- **1 次** 推荐信生成  

同一份「岗位描述」会被发送 **3 次**，简历会发送 **2 次**。

---

## 二、Token 消耗高的主要原因

### 1. 使用 GPT-4，单价高（最主要）

- 三个接口全部使用 **gpt-4**。
- GPT-4 的输入/输出单价远高于 **gpt-4o-mini**、**gpt-3.5-turbo**（约 10～30 倍）。
- 职位信息提取只是「公司名 + 职位名」的简单抽取，用 GPT-4 明显过剩。

### 2. 输入过长、未做裁剪

- **简历**：整份原文发送，可能 2000～8000+ 字。
- **岗位描述**：整段原文发送，可能 500～3000+ 字。
- 没有任何长度限制或摘要，长文档会直接拉高输入 token。

### 3. 提示词冗长

- 简历优化默认提示词：多段说明 + 6 条要求，可压缩。
- 推荐信默认提示词：8 条英文要求，可精简。
- 系统消息与用户消息存在重复（如「你是简历优化专家」说两遍），增加无效 token。

### 4. 同一内容多次发送

- 一次流程中，岗位描述在「提取职位信息」「优化简历」「生成推荐信」里各出现一次。
- 简历在「优化简历」和「生成推荐信」里各出现一次。
- 没有复用或摘要，重复内容重复计费。

### 5. 输出上限偏大

- 简历优化 `max_tokens: 2000`，在要求「直接返回简历」的前提下可以适当降低，减少为长输出预留的配额。

---

## 三、单次请求大致 Token 量（估算）

假设：简历 3000 字、岗位描述 1500 字、默认提示词。

| 功能 | 输入 token 估算 | 输出 token 估算 | 合计（约） |
|------|-----------------|-----------------|------------|
| 职位信息提取 | ~800 | ~50 | ~850 |
| 简历优化 | ~2500 | ~800 | ~3300 |
| 推荐信生成 | ~2500 | ~400 | ~2900 |
| **一次完整流程** | — | — | **约 7000+ token** |

且以上为 **GPT-4** 计费，若改用更便宜模型，同 token 量下费用可明显下降。

---

## 四、优化建议（按优先级）

1. **改用更便宜的模型（优先）**
   - 职位信息提取：改为 **gpt-4o-mini** 或 **gpt-3.5-turbo**。
   - 简历优化 / 推荐信：可改为 **gpt-4o-mini**，或在配置中提供「经济模式」选项。

2. **缩短默认提示词**
   - 合并系统消息与用户消息中的重复描述。
   - 用简短条款替代长列表，减少 100～300 输入 token。

3. **对输入做长度限制或截断**
   - 简历：例如只取前 N 字（如 4000 字）或前几段。
   - 岗位描述：例如只取前 M 字（如 2000 字），避免超长 JD 拉高 token。

4. **职位信息提取可前端先做简单解析**
   - 若岗位描述格式固定，可先用正则/简单规则提取公司名、职位名，失败再调 API，减少调用次数。

5. **可选：缓存与去重**
   - 相同岗位描述在短时间内不重复调用「职位信息提取」；相同「简历+岗位描述」不重复「优化/推荐信」（需定义缓存策略和过期时间）。

---

## 五、总结

| 原因 | 影响程度 | 优化难度 |
|------|----------|----------|
| 全部使用 GPT-4 | 高 | 低（改模型名/配置） |
| 输入未截断 | 高 | 低（加长度限制） |
| 提示词冗长 | 中 | 低（重写简短版） |
| 同一内容多次发送 | 中 | 中（需改流程或缓存） |
| 职位提取用 GPT-4 | 中 | 低（改为小模型） |

优先做：**模型切换** + **提示词精简** + **输入长度限制**，即可在不大改业务的前提下明显降低 token 消耗。

## 六、已实施的优化（代码层面）

当前项目已做如下修改以降低 token 消耗：

1. **模型**
   - 默认使用 **gpt-4o-mini**（主任务与轻量任务均可通过环境变量改回 gpt-4）。
   - 环境变量：
     - `OPENAI_MODEL`：简历优化、推荐信使用的模型（默认 `gpt-4o-mini`）。
     - `OPENAI_LIGHT_MODEL`：职位信息提取使用的模型（默认 `gpt-4o-mini`）。

2. **输入截断**
   - 简历：默认只取前 **4000 字**，可通过 `MAX_RESUME_CHARS` 修改。
   - 岗位描述：默认只取前 **2000 字**，可通过 `MAX_JD_CHARS` 修改。
   - 实现见 `server/utils/tokenHelper.js`。

3. **提示词**
   - 简历优化、推荐信、职位信息提取的默认提示词已缩短，减少重复与冗长说明。

若需最佳效果且不在意费用，可在 `.env` 中设置：
- `OPENAI_MODEL=gpt-4`
- `OPENAI_LIGHT_MODEL=gpt-4o-mini`（职位提取仍建议用便宜模型）
